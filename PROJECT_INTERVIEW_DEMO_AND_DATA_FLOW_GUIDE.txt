CARE SCHEDULE - INTERVIEW DEMO + DATA FLOW GUIDE (BEGINNER FRIENDLY)

Purpose of this file
- Help a totally new person test and explain the whole project.
- Give a clean interview demo sequence with talking points.
- Explain how data moves from frontend -> backend -> database and back.

--------------------------------------------------------------------------------
0) QUICK PROJECT OVERVIEW (SAY THIS IN INTERVIEW)
--------------------------------------------------------------------------------
This is a role-based healthcare scheduling and operations system with six portals:
- Admin
- Provider
- Patient
- FrontDesk
- Nurse
- Operations

Core capabilities:
- Master data management (sites, rooms, services, users, providers, holidays, rules)
- Appointment lifecycle (book -> check-in -> room assignment -> care -> completion -> outcome)
- Workforce planning (roster, on-call, leave impacts)
- Role-based authorization and audit traceability
- Soft-deactivation and active-only business validations

Architecture summary:
- Frontend: React + TypeScript + API modules per domain.
- Backend: ASP.NET Core Web API + services/repositories.
- Database: SQL Server via EF Core.
- Security: JWT auth + role checks on controllers/services.

DETAILED APPENDIX A - FRONTEND TO BACKEND DATA FLOW (TECHNICAL DEEP DIVE)
================================================================================

A1) FRONTEND LAYER BREAKDOWN
- Routing/Layouts:
  - Each role has dedicated route segments and layout wrappers.
  - Unauthorized routes are blocked in UI, but backend remains final authority.
- Page components:
  - Each page fetches data through `src/api/*` modules.
  - UI keeps local state for filters/search/forms and renders tables/cards.
- API modules:
  - Encapsulate axios calls, DTO types, and response unwrapping.
  - Keep endpoint paths and payload formatting out of page files.

Interview line:
"I separated page concerns from API concerns, so UI components stay readable and backend contracts stay centralized."

A2) BACKEND LAYER BREAKDOWN
- Controllers:
  - Expose HTTP endpoints and role-based authorization attributes.
  - Receive DTOs and return standardized API responses.
- Services:
  - Main business-rule engine.
  - Performs validation and orchestration across repositories.
- Repositories:
  - Data access against EF Core context.
  - Keep SQL/data concerns isolated from service logic.
- Database:
  - SQL Server schema with normalized domain entities.
  - Entity relationships enforce referential integrity.

Interview line:
"Controller = transport/security, Service = business policy, Repository = persistence."

A3) STANDARD REQUEST LIFE CYCLE (USE THIS TEMPLATE LIVE)
1. User action:
   - click button / submit form / open page
2. Frontend validation:
   - required checks, date/time guards, field-level constraints
3. API call:
   - axios request from domain API file with JWT header
4. Backend authorization:
   - role validation by `[Authorize]`
5. Business validation:
   - service checks status constraints, active-only references, conflicts
6. Persistence:
   - repository save/update in SQL Server
7. Response:
   - backend message + data DTO
8. UI state update:
   - success notice, table refresh, or inline error rendering

DETAILED APPENDIX C - DOMAIN ENTITIES AND WHY THEY MATTER
================================================================================

C1) Core master entities
- Site: location anchor for room, provider, scheduling context.
- Room: physical resource for appointment execution.
- Service: care offering type used across booking and provider mapping.
- User: identity and role root for all actors.
- Provider: clinical identity bound to a user.

C2) Scheduling entities
- Availability template: recurring weekly pattern.
- Availability block: one-off unavailability override.
- Slot: generated bookable inventory.
- Appointment: booked care event lifecycle record.
- Outcome: final clinical result attached to completed appointment.

C3) Operations entities
- Shift template: standard shift pattern.
- Roster assignment: who is assigned to which shift/day.
- On-call: backup emergency staffing.
- Leave request + leave impacts: continuity control.

C4) Frontdesk/nurse pipeline entities
- CheckIn: arrival event with token and current queue stage.
- Resource hold: prevents room double-booking for overlapping intervals.

Interview line:
"The data model mirrors real hospital flow: planning entities feed execution entities, and execution writes back operational outcomes."

DETAILED APPENDIX D - API CONTRACT AND ERROR-HANDLING STRATEGY
================================================================================

D1) Contract style
- Client sends typed DTO payloads.
- Backend returns wrapped response with `data` and `message`.
- UI shows backend message directly where useful.

D2) Validation layers
- UI validation:
  - improves user feedback speed
  - prevents obviously invalid calls
- Backend validation:
  - final source of truth
  - protects against malicious/manual calls

D3) Error categories to explain
- 400 Bad Request: business/validation rule failures.
- 401 Unauthorized: missing/invalid token.
- 403 Forbidden: role does not have access.
- 404 Not Found: missing resource (except designed nullable patterns).
- 409-like conflict behavior (if returned as validation message): duplicates/resource conflicts.

D4) Interview line
"We intentionally duplicate critical guards in backend even if frontend validates, because frontend is a convenience layer, not a trust boundary."

DETAILED APPENDIX G - DEMO TROUBLESHOOTING PLAYBOOK
================================================================================

If frontend list appears empty:
1. Check Network tab for status code.
2. Confirm correct role user is logged in.
3. Validate filters/search are not over-restrictive.
4. Check if test data exists for selected date/site.

If action fails with 400:
1. Read backend message in response.
2. Verify business preconditions (status/date/dependency).
3. Retry with valid entity state.

If action fails with 403:
1. Verify role.
2. Verify endpoint access is intended for that role.
3. Re-login if token is stale.

If compile/run issue due to dll lock:
1. Stop old API process.
2. Restart backend.
3. Retry build/run.
