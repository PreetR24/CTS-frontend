SECTION 1: PROJECT INTRODUCTION & HIGH-LEVEL
===============================================================================

Q1) What is your project about?
A) My project is a full-stack healthcare scheduling and operations platform called Care Schedule.  
It supports six role-based portals: Admin, Provider, Patient, FrontDesk, Nurse, and Operations.  
It handles complete lifecycle management:
- Admin configures master data and rules
- Provider creates availability and publishes slots
- Patient/FrontDesk books appointments
- FrontDesk handles check-in and room assignment
- Nurse continues patient flow
- Provider closes appointment with outcomes
- Operations manages roster/on-call/leave continuity
- Admin audits actions

Q2) What is the core business problem solved by your project?
A) It solves fragmented healthcare workflow management by connecting planning, appointment execution, staffing, and governance in one system.  
Instead of separate disconnected tools, this project enables end-to-end operational traceability from booking to outcome and staffing impact.

Q3) Why is this project technically significant?
A) It is significant because:
1. It has strict role-based access across all modules.
2. It enforces workflow integrity through status and business validations.
3. It prevents operational conflicts (slot overlaps, room conflicts, duplicate assignments).
4. It provides auditability and governance.
5. It includes both happy-path and invalid-case handling.

Q4) What technology stack did you use?
A)
- Frontend: React + TypeScript + Vite + React Router + Axios + React Hook Form
- UI: Tailwind utility classes + shared components + icons/charts
- Backend: ASP.NET Core Web API (.NET 10), service/repository architecture
- ORM/DB: EF Core + SQL Server
- Auth: JWT-based authentication + role authorization

Q5) Can you explain your architecture in one line?
A) Controller handles HTTP/authorization, Service handles business rules, Repository handles DB operations, and frontend consumes typed API modules with guarded routes.

SECTION 3: FRONTEND ARCHITECTURE QUESTIONS
===============================================================================

Q13) How is your frontend structured?
A) It is modular:
- `pages/` for role screens
- `layouts/` for role-level wrappers
- `routes/` for guarded routing
- `api/` for endpoint communication and DTO wrappers
- shared components in `components/`

Q14) Why did you use API modules instead of calling axios directly in each component?
A) API modules centralize endpoint paths, payload/response typing, and common response unwrapping.  
This reduces duplication and makes maintenance easier when backend contracts change.

Q15) How do you manage authentication in frontend?
A) JWT token is stored in local storage.  
Axios request interceptor attaches `Authorization: Bearer <token>`.  
Auth context fetches current user (`me`) and keeps role/landing page state.  
401 response triggers token cleanup and redirect to login.

Q16) How is role-based routing implemented in frontend?
A) I use route guards (`AdminGuard`, `ProviderGuard`, etc.) plus a parent `AuthGuard`.  
Routes are nested per role in `routes.ts`.  
This prevents unauthorized UI access and gives role-specific navigation.

Q17) Why is backend authorization still necessary if frontend has guards?
A) Because frontend can be bypassed.  
Real security must exist in backend with role checks.  
Frontend guards are UX-level convenience, backend is trust boundary.

Q18) How do you handle forms?
A) Mostly through React Hook Form:
- `register` for fields
- inline validations
- error display
- cleaner submit flow
This keeps forms scalable and reduces manual state handling.

Q19) Why React Hook Form instead of plain controlled inputs everywhere?
A) RHF gives better structure for complex forms with validations and fewer re-renders.  
It also improves consistency across multiple portals with many forms.

Q20) How do you handle UI filters/search?
A) I maintain filter states (`status`, `date`, `search text`, etc.) and apply them on fetched lists.  
In heavy list screens, memoized computations are used to avoid recomputation overhead.

Q21) How do you show backend errors in UI?
A) API call failures catch backend message and render user-friendly notices inline or near actions.  
This avoids generic “something went wrong” and helps users recover.

Q22) How do you manage empty data states?
A) Each list/table/card view has explicit empty-state UI (“No records found”) so user sees a meaningful state, not blank screen.

SECTION 4: BACKEND ARCHITECTURE QUESTIONS
===============================================================================

Q23) Why did you choose Controller-Service-Repository architecture?
A) It keeps concerns clean:
- Controller: transport + authorization
- Service: business logic
- Repository: data access
This structure improves testability, readability, and long-term maintainability.

Q24) What role do DTOs play in your project?
A) DTOs decouple external API contracts from internal entities.  
This prevents direct entity exposure and allows controlled request/response shaping.

Q25) How does dependency injection help your backend?
A) Controllers/services depend on interfaces, not concrete classes.  
This improves modularity, allows easier replacement/mocking, and follows DIP principle.

Q26) How is EF Core used?
A) EF Core is used with SQL Server through `DbContext`.  
Entities are mapped with fluent configuration for keys, defaults, constraints, indices, and relationships.

Q27) Did you use migrations?
A) Yes, schema is migration-driven in infrastructure layer, enabling reproducible database setup and versioned schema changes.

Q28) How do you maintain consistent API responses?
A) I use a standardized `ApiResponse` contract and a global exception middleware to normalize errors and messages.

Q29) What does your global exception middleware do?
A) It maps exceptions to appropriate HTTP statuses and business-friendly error messages:
- `ArgumentException` -> 400
- `KeyNotFoundException` -> 404
- SQL duplicate/fk constraint violations -> conflict-style responses
- unknown exceptions -> 500

Q30) Why is central exception handling better than per-controller try/catch?
A) It avoids repeated boilerplate, gives consistent behavior, and makes error contracts predictable across the API.

SECTION 5: AUTHENTICATION & AUTHORIZATION QUESTIONS
===============================================================================

Q31) How does login work in your project?
A) User submits credentials -> backend validates -> JWT token returned -> frontend stores token -> future API calls carry bearer token.

Q32) What claims are important in your token?
A) Role claim is most important for endpoint authorization.  
User identity claims help in ownership checks for self-only data operations.

Q33) How do you prevent users from accessing other role APIs?
A) Controllers use `[Authorize(Roles = "...")]`.  
Even if frontend route is manually changed, backend denies access (403).

Q34) Do you only trust role from frontend request body/query?
A) No. Role is trusted from authenticated token claims.  
Frontend-sent role can be used for UX but security decisions are backend-claim-based.

Q35) How do you handle unauthorized and forbidden states in UI?
A) 401 clears token and redirects login; 403 shows permission message without forced logout.

SECTION 6: DATA INTEGRITY & BUSINESS RULE QUESTIONS
===============================================================================

Q36) What are examples of strong business validations in your project?
A)
- Past-date block creation blocked
- Past-date on-call blocked
- Outcome allowed only after appointment completion
- Duplicate provider-service mapping blocked
- Duplicate shift assignment blocked
- Room-time conflict blocked on assignment
- Leave approval depends on impact resolution

Q37) Explain active-only validation pattern.
A) Whenever one entity references another (FK-like business reference), system checks referenced entity is active.  
This avoids linking appointments/schedules/configuration to deactivated data.

Q38) How do you handle deactivation vs deletion?
A) Most business entities use status-driven soft deactivation.  
This preserves historical records and prevents broken references.

Q39) How do you prevent duplicate check-ins?
A) Check-in table has unique appointment linkage and service-level check verifies no existing check-in before creating new one.

Q40) How do you ensure room isn’t assigned to two appointments at same time?
A) Backend checks overlap against:
1) existing appointment room usage for date/time
2) active resource holds for date/time
If conflict exists, assignment is rejected.

Q41) How do you ensure outcome flow is correct?
A) Outcome creation is guarded by appointment status.  
Only completed appointment can receive outcome; UI also reflects this by conditional button visibility.

Q42) How do you handle leave impacts?
A) Leave submission/approval logic generates or validates impacts.  
Operations must resolve impacts before approval in constrained scenarios.

SECTION 7: SLOT, SCHEDULING, AND APPOINTMENT FLOW QUESTIONS
===============================================================================

Q43) Explain availability template and block difference.
A) Template defines recurring weekly availability windows.  
Block defines one-off unavailability interval (specific date/time) overriding template-generated slots.

Q44) How are slots generated?
A) Slot generation reads provider templates, computes date-time windows, applies duration, and excludes overlaps with active blocks/blackouts/conflicts.

Q45) How do you ensure blocked time doesn’t generate slots?
A) Slot generation logic filters candidate intervals by checking overlap with active blocks for that day before persisting slots.

Q46) How do appointments connect with published slots?
A) Appointment booking consumes published slot availability and updates appointment records with provider/site/service/date/time.

Q47) What appointment statuses are relevant in your flow?
A) Commonly:
- Booked
- CheckedIn
- RoomAssigned/InRoom/InProgress (depending on stage mapping)
- Completed
- Cancelled/NoShow variants where applicable

Q48) Why is strict status transition important?
A) It prevents invalid workflow jumps and preserves operational truth across roles.

Q49) How is patient outcome shown in patient portal?
A) Patient appointments fetch outcome by appointment for completed bookings; if outcome exists, it is shown in UI.

Q50) How did you handle “outcome missing” API gracefully?
A) Endpoint supports nullable outcome response path, so frontend can show “not available yet” without hard 404 UX break.

SECTION 8: FRONTDESK-SPECIFIC QUESTIONS
===============================================================================

Q51) How does waitlist continuation work?
A) Waitlist entry can push context (patient/provider/service/site/date) into booking flow and redirect user to continue booking from that point.

Q52) Why is this useful?
A) It avoids re-entering data manually and accelerates frontdesk operations during high traffic.

Q53) How is check-in token handled?
A) Token is generated/persisted on check-in (if not provided) and surfaced in queue/check-in UI for operational tracking.

Q54) Why not rely on visual index number instead of token?
A) Visual index changes with filtering/sorting; persisted token is stable operational identity for that check-in event.

Q55) What happens when FrontDesk assigns a room?
A) Backend updates check-in status/room and syncs appointment room/status as needed, while applying conflict protections.

Q62) Which design principles did you apply?
A)
- Separation of Concerns
- Single Responsibility
- Dependency Inversion (via interfaces + DI)
- DRY (shared API wrappers/components/middleware)
- Defense in Depth (frontend + backend authorization)
- Fail-fast validation

Q63) Which design patterns are used?
A)
- Repository pattern
- Service layer pattern
- Dependency injection pattern
- Middleware pattern
- Guard pattern (routes + authorization)
- Context provider pattern (frontend auth)
- DTO mapping pattern

Q64) Why did you not keep all logic in controllers?
A) Controller-heavy code becomes hard to test and maintain.  
Business logic belongs in service layer for reusability and cleaner API endpoints.

Q65) Why did you not use entity models directly in frontend contracts?
A) DTOs let us control payload shape, avoid leaking internal schema, and evolve backend independently.

SECTION 11: DATABASE & EF CORE QUESTIONS
===============================================================================

Q66) How are relationships modeled?
A) Through EF fluent configuration:
- one-to-one (e.g., appointment-checkin/outcome)
- one-to-many (e.g., site-rooms, user-notifications)
- proper delete behaviors and foreign key constraints.

Q67) Why are unique constraints important?
A) They prevent duplicate domain records at database level even if application checks are bypassed.

Q68) Why use default values at DB level?
A) It guarantees baseline status/timestamps even if callers miss fields.

Q69) How do you handle DB constraint errors in API?
A) Global middleware maps SQL exception codes into readable conflict messages (duplicate/fk constraints).

Q70) What is the advantage of EF Core fluent config over only data annotations?
A) Fluent API gives centralized, expressive control for advanced indexing, relationships, and behaviors.
