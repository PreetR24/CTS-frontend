B) Admin portal full test

1. Dashboard

- Validate role counts are non-zero from seed data.

------------------------------------

------------------------------------
I) Final full project business flow
------------------------------------
1) Admin configures sites/rooms/services/provider mappings.
2) Provider sets availability and blocks; publishes slots.
3) Patient books appointment OR FrontDesk books/waitlists.
4) FrontDesk checks in patient and assigns room.
5) Nurse continues queue statuses until provider consultation completes.
6) Provider marks appointment outcome.
7) Operations manages roster, on-call, leave impacts, and staffing continuity.
8) Admin audits actions and verifies governance/config.

------------------------------------------------------------
J) Edge-case and validation stress testing (must run)
------------------------------------------------------------

Run this section after normal flow. This is the high-priority evaluation part.

1) Field-level validation checks (every form in every role)
- Required fields: submit empty form and verify inline error + no API call.
- Min/max checks: test below min and above max numeric values.
- Length checks: test 1-char and over-limit strings.
- Pattern checks: invalid email/phone/time/date formats.
- Date checks:
  - past date when not allowed
  - end date before start date
  - same date boundary behavior
- Time checks:
  - end time <= start time
  - overlap with existing slot/template/block
- Dropdown checks:
  - keep default “Select …” and submit
  - stale dependent option after changing parent dropdown
- Duplicate checks:
  - duplicate site/service/holiday/template/provider-service mapping where uniqueness expected.

2) API/error contract checks (all forms)
- Trigger backend validation errors and verify exact message is visible in UI.
- Trigger forbidden action (403) and verify user-friendly message.
- Trigger not found (404) by editing/deleting stale record and verify graceful handling.
- Trigger conflict/duplicate and verify clear error toast/text.
- Disable internet briefly or stop backend:
  - verify network error message shown
  - verify page does not crash and retry path exists.

3) Soft-delete/reactivation checks
- For each module using soft delete, verify:
  - delete changes status only (not hard removed in DB)
  - inactive filter shows record
  - activate restores usability
  - linked constraints show meaningful backend reason on blocked deactivation.

4) Role-specific negative authorization checks
- Login as each role and attempt restricted pages and actions:
  - URL direct navigation
  - button/API action from DevTools replay
- Verify unauthorized requests are blocked and do not mutate DB.

5) Concurrency/race checks (critical)
- Open same entity in 2 tabs:
  - edit in tab A, delete/deactivate in tab B, then save in tab A.
  - verify stale update error is handled cleanly.
- Double-click submit buttons quickly:
  - verify no duplicate create rows.
- Assign same room or same shift concurrently:
  - verify backend conflict protection and UI error feedback.

---------------------------------------------------------
K) Role-wise edge checklist for evaluator execution
---------------------------------------------------------

Admin edge checklist
- Site deactivation with active dependencies -> blocked with reason.
- Duplicate holiday (same site/date) -> blocked.
- Invalid capacity/SLA values -> blocked.
- User create invalid email/phone -> blocked.
- User self-edit role mutation attempt -> blocked.

Provider edge checklist
- Template duplicate same day/time -> blocked.
- Block overlapping template -> handled correctly.
- Leave request in past date range -> blocked.
- Appointment status invalid transition -> blocked with backend message.
- Provider service duplicate mapping -> blocked.
- Provider service mapping with inactive provider/service -> blocked.

Patient edge checklist
- Booking with missing dependent dropdown selection -> blocked.
- Booking past date/time -> blocked.
- Booking when no slots available -> proper empty-state and no crash.
- Profile invalid fields -> inline errors shown.
- Booking with inactive site/provider/service/patient or inactive provider-service mapping -> blocked.

FrontDesk edge checklist
- Booking with missing provider/service/site/date -> blocked.
- Waitlist to booking continuation with stale/invalid params -> graceful error.
- Room assignment conflict (already held room/time) -> blocked.
- Check-in/queue actions on invalid appointment status -> blocked.
- Waitlist create with inactive site/provider/service/patient -> blocked.

Nurse edge checklist
- Schedule fetch unauthorized for other user -> blocked by backend (self-only).
- Queue action invalid next-status transition -> blocked with message.
- Empty room/queue datasets show proper no-data messages.

Operations edge checklist
- Shift template duplicate (same site/role/name/time) -> blocked.
- Assignment role mismatch (template role vs selected staff role) -> blocked.
- Edit assignment replacement cross-role attempt -> not possible in dropdown; backend also blocks if forced.
- On-call backup same as primary -> blocked.
- Leave approval with unresolved impacts -> blocked.
- Weekly calendar role split should match DB (`Nurse` + `FrontDesk` counts).

Completion criteria
-------------------
- No blocking console errors in critical pages.
- CRUD works with backend persistence.
- Soft delete/reactivation works where designed.
- Role restrictions hold.
- Status transitions reflect backend truth.
- Validation errors are shown for invalid input in every tested form.
- Backend errors (400/403/404/409/500/network) are surfaced with clear user-facing messages.
