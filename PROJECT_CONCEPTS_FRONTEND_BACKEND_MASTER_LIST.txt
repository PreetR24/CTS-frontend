C) CROSS-CUTTING DESIGN PRINCIPLES USED
================================================================================

1) Separation of Concerns (SoC)
- Frontend pages vs API modules vs backend controllers/services/repos.
- Where used:
  - `frontend/src/app/pages/**` vs `frontend/src/api/**`
  - `backend/API` vs `Services` vs `Repositories`

2) Single Responsibility Principle (SRP)
- Each service/repository focused on one domain area.
- Where used:
  - e.g., `CheckInService`, `LeaveService`, `RosterService`, etc.

3) Dependency Inversion Principle (DIP)
- High-level modules depend on interfaces, not concrete classes.
- Where used:
  - constructor injection across services/controllers
  - interface folders under services/repositories

4) Open/Closed Style Extension
- New domain modules added via new service/repo/controller without rewriting whole stack.
- Where used:
  - separate modules for waitlist, checkin, outcome, roster, on-call, etc.

5) DRY (Don’t Repeat Yourself)
- Shared response wrappers, auth context, axios interceptors, common UI components.
- Where used:
  - `ApiResponse`, `axios.ts`, `AuthContext.tsx`, shared UI component folders

6) KISS (Keep It Simple, Simple Hooks + Modular APIs)
- Emphasis on approachable React patterns and modular APIs.
- Where used:
  - broad use of `useState/useEffect` + per-domain API files

7) Fail-fast Validation
- Reject invalid state early in service layer.
- Where used:
  - guards in services (past dates, wrong status, duplicates, conflicts)

8) Defense in Depth (Security)
- Frontend route guard + backend role enforcement + JWT checks.
- Where used:
  - frontend guard files + backend `[Authorize]` attributes

9) Explicit Error Contracts
- Predictable errors and messages from middleware + UI rendering.
- Where used:
  - `GlobalExceptionMiddleware.cs`
  - frontend error handling in page action handlers

E) MINOR/BUT-IMPORTANT TECHNICAL CONCEPTS
================================================================================

1) Promise-based async programming (`async/await`) in frontend/backed.
2) Parallel I/O with `Promise.all` in frontend page initialization.
3) `Map` usage for O(1) lookup of names by ID in UI lists.
4) Responsive UI with utility classes (grid breakpoints and adaptive layout).
5) Empty-state and fallback UX as resilience pattern.
6) String normalization for status handling (e.g., queue status mapping).
7) Local-only environment override file for secure local config (`appsettings.Local.json`).
8) Database-level defaults (`HasDefaultValue`, `HasDefaultValueSql`) for stable inserts.
9) One-to-one linkage for check-in/outcome uniqueness per appointment.
10) Conflict-safe room assignment using overlap checking logic.

F) INTERVIEW CHEAT LINE (DIRECT SPEAK)
================================================================================

"This project uses a layered, interface-driven architecture with clear separation between UI, API transport, business rules, and persistence. On frontend, we use React with guarded role routing, API abstraction, and form validation with React Hook Form. On backend, we use ASP.NET Core with JWT auth, role authorization, service/repository patterns, EF Core SQL Server persistence, and global exception middleware for consistent error contracts. The system emphasizes workflow integrity through status-based validations, active-only constraints, conflict checks, and auditable operations."
